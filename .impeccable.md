# Impeccable Design Context for MergeChain

## Design Context

### Users
Developers and engineering teams working with stacked or interdependent pull requests on GitHub. Context includes code review, feature branching in complex codebases, and increasingly, workflows where AI coding assistants (Claude, Cursor, Copilot, Grok, etc.) generate code and open multiple PRs in a single session. They need reliable enforcement to prevent out-of-order merges without adding backend complexity or changing their GitHub experience.

Key jobs to be done:
- Declare "this PR is blocked by #123" (or cross-repo) directly in the GitHub UI.
- Have AI agents automatically manage dependencies when creating stacked PRs.
- See clear visual feedback (blocked merge button, chains, reverse "depends on this" views) without leaving GitHub.
- Install and use with minimal friction (store, direct ZIP, or source).

### Brand Personality
Honest, practical, grounded, no hype. "A strong nudge and a shared source of truth." Technical but approachable and reliable. Emphasizes:
- Simplicity: no backend, data lives in the PR itself, token never leaves browser.
- Complementarity: works in native GitHub UI, with existing workflows, and alongside AI tools.
- Trust: free, transparent (honest caveat about client-side enforcement), not affiliated with GitHub.
- Precision: enforces merge order for stacked PRs like GitLab, with transitive support and cycle detection.

Tone should feel like a well-engineered tool: clear, functional, slightly opinionated, focused on solving a real pain without unnecessary flourish. Avoid corporate marketing speak or "AI magic" vibes.

### Aesthetic Direction
Clean, technical, developer-focused dark theme. GitHub-native but elevated with refined details and subtle craft. 

- **Theme**: Dark (for focus during deep work, code review, late-night coding). Use GitHub-inspired dark palette but with intentional refinements.
- **Typography**: System fonts for familiarity and performance (as it's a lightweight static site + extension UI), but with clear hierarchy, good contrast, and monospace for code/references. Avoid overly "designy" display fonts.
- **Visual language**: Precise, no-nonsense. Functional demo that mirrors the real extension UI. Use grids for clarity, but vary spacing for rhythm. Subtle borders, clear state colors (blocked red, ready green, accent blue). 
- **References**: GitHub's own UI (for seamlessness), high-quality technical documentation sites, precision engineering tools. Anti-references: overly glossy AI tool sites, neon cyberpunk, generic card-heavy marketing pages, heavy gradients or glassmorphism.
- **Differentiation**: The combination of browser enforcement + seamless AI agent skill support + zero backend. The site should feel as reliable and practical as the product. The interactive demo is central and should feel "real."

Constraints:
- Static single-file HTML/CSS/JS for the landing (simple deployment).
- Must work alongside the Chrome extension's own UI (popup, settings – also dark, system fonts).
- Performance: fast load, no heavy assets.
- Accessibility: good contrast, keyboard nav, screen reader friendly.
- Cross-platform: looks good in Chrome (primary), but general web.

### Design Principles
1. **Honesty and clarity first**: Every element should communicate exactly what the product does and its limitations. No overpromising.
2. **Seamless with GitHub + AI workflows**: The design should feel native to GitHub while highlighting how it integrates with AI agents without friction.
3. **Functional beauty**: Prioritize usable, scannable interfaces. Polish details (spacing, hierarchy, states) but keep it lightweight.
4. **Practical for power users**: Assume users understand stacked PRs, Git, and agents. Provide direct paths (store, ZIP download, skill prompt) without hand-holding fluff.
5. **Memorable through utility, not gimmicks**: The demo and clear AI integration should be what people remember, not decorative elements.
